## 设计原则
| 原则 | 做法 |
|---|---|
| 小白开箱即用 | 上传后自动分析，默认生成学习版 PDF |
| Let it crash | 对真实能力边界明确标注，不做虚假承诺 |
| 第一性原理 | 目标不是保留动画，而是把现场讲解顺序变成可读讲义 |

## V2 开发发现
| 发现 | 处理 |
|---|---|
| `.pptx` 本质是 ZIP + OOXML，首版直接解析 slide XML 比截图猜测更严谨 | 实现 `pptx_parser.py`，读取对象、备注、动画目标 |
| 基础动画可以从 `p:timing` 下的 `animEffect` 和 `spTgt` 建立顺序 | V2 支持 `fade`、`wipe`、`appear`，其他动画写入 warning |
| 遮挡检测不能猜语义，只能基于边界框和 z-order 给风险提示 | 实现 top-layer occlusion warning，不自动改写原内容 |
| Python `tempfile.TemporaryDirectory` 在当前 Windows 环境下会创建权限异常目录 | 测试改用项目内 `workspace/test_runs` 普通目录 |
| 沙箱内调用 Edge 生成 PDF 失败 | CLI 端到端验证使用放行模式运行 |
| PowerShell `Start-Process` 受 Path/PATH 环境冲突影响 | 本地服务最终用 .NET ProcessStartInfo 放行启动 |

## 开源方案调研
| 方案 | 发现 | 决策 |
|---|---|---|
| LibreOffice headless | 可直接用 `--headless --convert-to pdf` 做文档转 PDF | 作为原生保真底座 |
| Unoserver | 官方 README 说明它把 LibreOffice 作为常驻转换服务，并推荐替代 unoconv | 服务化转换首选 |
| unoconv | 仓库已归档，README 说明已 deprecated，推荐 Unoserver | 不作为新核心依赖 |
| Collabora Online | 开源、基于 LibreOffice，可在浏览器查看/编辑文档，也适合容器化转换 | 作为生产化备选，不作为本地 demo 首选 |
| python-pptx | slide XML 里有 `spTree` 和 `timing`，适合读取对象与动画结构 | 继续用于诊断，不用于版式重建 |
| Apache POI XSLF | Java 里有 XSLFSlide、XSLFPictureShape、XSLFTextShape 等对象模型 | 仅备选，当前不切技术栈 |
| PPspliT | 把动画步骤拆成静态 slide 序列，目标就是方便 PDF/打印 | 借鉴“拆动画为增补页”的产品思路 |

## 路线纠偏
| 纠偏点 | 结论 |
|---|---|
| 用户反馈“现在只是扫描文字” | 认可，当前 V2 只适合证明解析链路，不适合作为最终转换方案 |
| PPT 转 PDF 的基础体验 | 必须像普通转换器一样保留排版和图片 |
| 我们的创新点 | 不替代原生转换，而是在问题页旁边追加展开页、解释页和留白页 |
| 本机环境 | 当前机器未检测到 LibreOffice，需要后续安装或用容器/服务端方案 |

## 动画页设计补充
| 用户想法 | 设计结论 |
|---|---|
| 有动画的页面如果每次细小变化都加增强页会繁琐 | 引入“页数预算”，按变化量和语义组规划页数 |
| 希望尽量还原原生意思，但允许二次排版 | 学习版 PDF 可在同页保留核心图片/图表并加说明栏、编号、高亮、留白 |
| 希望提示用户这页 PPT 变了什么 | 使用变化编号、提示条、局部放大、幽灵层、阅读顺序线 |
| 不希望机械按动画数量拆页 | 按语义分组，复杂页最多建议 2-4 页 |

## 双模式决策
| 决策 | 结论 |
|---|---|
| 是否允许 study.pdf 替换原生页为二次排版页 | 允许，但必须保留 `base.pdf` 作为原生对照版 |
| 基础版是否必须接入 AI API | 不需要 |

## 逐项确认记录
| 项 | 结论 |
|---|---|
| 产品一句话定义 | 确认：面向课程课件和企业培训材料，将 PPT/PPTX 转换为适合阅读、批注和复习的 PDF |
| 动画导读呈现原则 | 动画变化要形象地融入 PDF 页面本身，不只是简单放在旁边说明 |
| 用户和使用场景 | 确认：大学生课后复习、企业员工培训回看、老师/培训负责人发放和沉淀材料 |
| 输入文件范围 | 确认：首版只支持 `.pptx`；`.ppt`、PDF、Keynote、加密文件暂不支持；音视频只保留静态页并提示 |
| 输出文件类型 | 确认：首版输出 `base.pdf`、`guide.pdf`、`report.json`，保留 `preview.html` 用于 Web 预览；`ai_notes.pdf` 后续再做 |
| 输出模式定义 | 确认：首版不做单独批注版；只保留标准版和动画导读版，核心是研究动画如何融入 PDF |
| 动画融入策略 | 确认：原生增强、融合重排、少量拆分页三种形态；融合重排是核心攻关点 |
| 重排攻关目标 | 在原意不变的前提下，让改后排版协调、易懂、能体现动画顺序 |
| 同页增强和融合重排视觉规则 | 确认：标注不小于正文 80%；一种强调色加灰色辅助；统一 ①②③；箭头只表示阅读顺序/流程；高亮只圈变化对象；每个变化点最多一句话 |
| 二次排版触发条件和底线 | 确认：只在遮挡、弹出层叠加、讲解顺序丢失、信息太密、多逻辑组等情况下重排；不改结论、不删核心图文、不凭空增加知识、保留 `base.pdf` |
| 拥挤判断 | 标记为算法关键，需要判断页面信息密度、对象重叠、可读空间和标注可放置性 |
| 新增页规则 | 确认：能同页融入则不新增；标注遮挡核心内容、多逻辑组、步骤超过 5 个且无法合并、明显遮挡、原页拥挤时新增；单页最多 4 页，极复杂页提示人工拆课件 |
| AI 接入边界 | 确认：基础版不依赖 AI；AI 版解释知识点、动画意图和复习提示；必须可关闭、标来源、依据不足写需补充、禁止无依据扩写 |
| 技术架构 | 确认：上传服务、原生转换器、PPTX 解析器、页面分析器、重排决策器、PDF 增强器、AI 解释器、前端 |
| 开箱即用封装 | 确认：最终需要面向小白封装，不能要求用户走代码流程；目标是一键启动/安装包/本地网页式工具 |
| 不支持边界 | 确认：首版不支持 `.ppt`、Keynote、加密文件、宏、音视频嵌入、复杂路径动画、点击触发/交互动画；但后续会考虑音视频、复杂路径动画、交互动画的文字还原和理解性处理 |
| 开发阶段和验收标准 | 确认：V3A 原生转换、V3B 动画分析、V3C 动画导读页、V3D 重排决策、V3E 前端封装、V3F 样例验证 |

## V3D 开发发现
| 发现 | 处理 |
|---|---|
| 动画融入 PDF 不能只靠追加说明页，必须先在原生页本身给出视觉锚点 | `augment_plan.json` 新增 `inline_markers`，记录动画目标的顺序、文本、类型和边界框 |

| 复杂重排不能一口吃完 | V3D 先完成轻量同页增强；融合重排、说明栏避让、拥挤布局决策继续作为后续攻关 |
| Codex 沙箱内启动的后台服务会在命令结束后被回收 | 本地持久预览服务需要用沙箱外 `pythonw` 启动，并用 `/api/health` 复查 |
| `guide测试.pdf` 在复杂课件上出现大量绿色框，用户无法理解其意义 | 这不是可忽略问题；绿色框当前只是调试式视觉锚点，违反“动画形象融入 PDF 页面本身”的目标，下一步应改为少量编号徽标、弱高亮或说明栏联动 |
| `guide测试.pdf` 从 42 页源 PPT 变成 75 页，后半段导读页成堆且文本溢出 | 复杂页导读页生成策略过粗，不能继续机械对所有 `expand_after_native` 页追加说明页；需要先做页数预算上限、分组摘要和可读版式 |
| `Review+chapter24-27.pptx` 是更真实的复杂排版样例 | 已确定为后续 V3D/V3F 的重点回归样例，用来测试拥挤判断、标注避让和复杂页处理 |
| Review 样例止损热修验证 | `report_only` 策略可先保护可读性：复杂/不支持/过多动画页不硬生成导读页，`guide.pdf` 页数回到 42 页 |
| 单纯编号仍然不够解释动画变化 | 编号旁需要短提示；当前按 `first_change`、`covered_content`、`key_result` 生成“先出现 / 遮挡变化 / 关键结果”，降低用户理解成本 |
| 遮挡类标注不能靠文案猜测 | 先用对象边界框和 z-order 判断动画对象是否覆盖已有对象，作为 `covered_content` 的结构化依据 |
| 标注可放置性应先看边缘空白带 | 只在上下左右存在足够空白时放提示条；无空白时不生成 `inline_markers`，避免再次遮挡正文。 |
