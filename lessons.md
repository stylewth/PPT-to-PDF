# Lessons

| 日期 | 经验 |
|---|---|
| 2026-05-16 | Web 服务是长驻进程，后端代码改完必须重启 8765 并用 `/api/convert` 重新生成样例，再检查 `augment_plan.json` 的真实策略；不能拿旧 Web job 或旧下载 PDF 判断新算法效果。 |
| 2026-05-17 | OLE 公式类 `graphicFrame` 不只要改外层 `p:xfrm`，还要同步更新内层 fallback `a:xfrm`；LibreOffice 转 PDF 可能渲染 fallback 图，否则公式看起来仍在旧位置。 |
| 2026-05-16 | 重排编号不能贴在对象左上角直接盖图文；应按对象外侧候选位避让目标 bbox 和既有对象。公式类 `graphicFrame` 不能缩放，否则 LibreOffice 转 PDF 容易压缩或偏移，应默认只平移保尺寸。 |
| 2026-05-16 | 对象级重排不能把图、公式、正文当成独立盒子排序；必须先识别图文对应组，相关视觉对象跟随对应正文就近移动，并在组内避让，避免公式离开语义位置或反向遮挡。 |
| 2026-05-16 | 无直接遮挡关系的图片、公式、图形应默认保持原位；严重页面扩展重排只能自动纳入文本对象，视觉对象只有进入遮挡边或动画覆盖关系时才移动。 |
| 2026-05-16 | 对象级重排必须统计真实 `operations`，不能把无操作页面包装成重排成果；文本框重排时不能压缩高度，正文列也要尽量保留原宽度，否则 PowerPoint 会发生文本外溢并制造新的遮挡。 |
| 2026-05-13 | V3D 的动画标注不能停留在调试框或纯编号；复杂课件优先保护原生可读性，简单页才做少量、语义明确、不会压住正文的页内提示。 |
| 2026-05-14 | “不支持动画”不等于“没有动画”；必须区分已解析但未能严谨解释的 OOXML 动画节点。能从 `animEffect filter` 明确识别的效果可以扩展，裸 `anim` 数值动画不能硬猜。 |
| 2026-05-14 | 裸 `anim` 只有在 `attrName` 明确是 `ppt_x/ppt_y` 且起止值发生变化时才可识别为位置移动；无变化的坐标辅助节点应跳过，避免把同一次移动拆成无意义步骤。 |
| 2026-05-14 | 用户明确否定“多加导读页”的主路线；复杂动画页的价值应来自替换式融合重排，`base.pdf` 保留原生对照，`guide.pdf` 直接变得更易读，而不是追加旁白页。 |
| 2026-05-14 | 用户进一步明确：重排不是抽出内容重画，而是在保留 PPT 原有画面基础上微调；路线应转向“LibreOffice 原生 PDF + PDF 层局部遮盖、搬移、缩放、叠加”。 |
| 2026-05-16 | 遮挡处理的目标不是只标“有遮挡”，而是让被遮挡内容在 PDF 中换到清晰位置可见，并用编号、箭头或关系线表达前后流程关系。 |
| 2026-05-16 | 遮挡内容不固定放到展开栏；应先利用原页面空白区，再考虑缩放让位或独立展开区。排版必须多轮优化，同时满足可解释的科学性和视觉美观。 |
| 2026-05-16 | “保留 PPT 基本页面”不等于“原 PDF 主体不动”。如果原生最终态本身已经文字/公式重叠，学习版必须移动或缩放具体 PPT 对象，让主体区域变清楚；只在侧栏复制遮挡前后区域属于路线偏移。 |
| 2026-05-17 | `graphicFrame` 公式仅同步 OOXML 坐标仍不够；LibreOffice 可能裁剪 OLE 预览或带入原位置文字。最终 PDF 应优先从 PPTX fallback 图片提取公式预览，白边透明裁剪后按目标 bbox 覆盖，缺失预览时才裁剪 `base.pdf` 区域。 |
| 2026-05-17 | 公式和图像的组内避让不能容忍小比例压边；已经移动的视觉对象也必须进入后续对象的占位表。空间不足时可对高分辨率公式预览小幅缩放，优先保证不遮挡和可读性。 |
| 2026-05-17 | 重排不能把正文强行压成左栏再把所有图片放右侧；应先保留正文组原有水平意图，再在局部空白中按图文关系放公式和图片，右侧只是一种候选位置。 |
| 2026-05-17 | 避免右栏化后仍可能变成散点式重排；后续必须以语义组局部修复为主，图文关系和流程关系优先于全局不重叠。 |
| 2026-05-17 | 后续重排必须先通过路线契约：局部修复、语义组不散、稳定对象不动、截图人工验收；几何不重叠不能单独算成功。 |
| 2026-05-17 | 路线防偏不能只靠“看起来不重叠”的几何指标；必须同时检查语义组是否仍靠近、稳定对象是否被误动、多个视觉对象是否被系统性推成右栏/左栏。每次后端排版改动后都要重新转换 `test.pptx` 并截图验收，不能用旧 job 或旧截图判断。 |
| 2026-05-18 | 重排关系线不能把箭头端点贴到目标 bbox 边界；PowerPoint/PDF 渲染箭头会占用额外长度，必须给目标对象外侧留出箭头安全距离，避免压住公式或图像。 |
| 2026-05-18 | PPTX 里的 GIF 可能只是普通图片关系，不能只找 video/media 标签；应从 `a:blip` 关系目标后缀识别动态媒体，并把原始对象 bbox 写进媒体清单。 |
| 2026-05-18 | PPTX 视频对象常带一张封面图和一个 `a:videoFile` 关系；解析媒体时应优先读 `a:videoFile`/`a:audioFile`，否则会把视频误判成普通封面图片。 |
| 2026-05-18 | GIF 转 PDF 不应假装保留动画播放；更符合学习版目标的是保留原位静态画面，导出原始 GIF，并在同页空白区放关键帧条说明动态过程。 |
| 2026-05-18 | 媒体关键帧条不能只避让媒体本体，还要避让同页标题、文字、图像等占位对象；否则会出现“解决 GIF 可读性，却压住原页面信息”的路线偏移。 |
| 2026-05-18 | 对动态媒体，原 PPT 放置的大封面区域本身就是媒体语义位置；学习版 PDF 应优先把该区域替换为关键帧宫格，而不是额外占用页面空白放摘要条。 |
| 2026-05-18 | Web 报 `No module named 'fitz'` 时先查服务进程实际 Python；命令行环境有 PyMuPDF 不代表 Web 服务环境也有，必须给服务正在用的解释器安装依赖并重启。 |
| 2026-05-19 | GIF 关键帧宫格不能永远锁死在原图 bbox；小媒体应以原 bbox 为锚点向附近空白扩展，同时避让标题、正文和其它占位对象，避免“原位但不可读”。 |
| 2026-05-19 | Windows 弹出 VS Code `d3dcompiler_47.dll` DeleteFile 失败时，先看路径归属；路径在 VS Code 安装目录且转换接口仍成功，不能误判为 PPT/PDF 转换链路错误。 |
| 2026-05-19 | 关闭 VS Code 相关锁进程后要重新检查 8765 服务；如果后端原本跑在 VS Code 终端里，窗口关闭会顺带停掉转换服务，必须重启服务并用 `/api/convert` 真实转换验证。 |
| 2026-05-19 | 小 GIF 周围的 PPT 文本框可能比真实文字宽很多，不能因为硬避让文本框就退回原小框；原位扩展失败时，应改用紧邻空白区放大关键帧，保留媒体语义位置同时提升可读性。 |
| 2026-05-19 | 动态媒体学习版的默认语义是“关键帧序列替换原媒体”，不是“保留封面再额外补图”；空间不足时可把第 1 帧放回原媒体区，其余帧排到附近空白区，形成同一时间序列。 |
| 2026-05-20 | “打分优化”必须落到可审计的候选评分和截图验收；GIF 关键帧不能保留原大图当背景，应清掉原媒体区、按 source_frame 与 split_panel 评分放置，并只恢复真正受影响的文字，避免重绘造成重复或误覆盖。 |
| 2026-05-20 | 大 GIF 的关键帧宫格是替换原媒体区，不是叠加在封面上；只要进入关键帧绘制，必须先清完整 source_rect，再绘制评分选出的宫格，避免原封面从面板边缘露出。 |
| 2026-05-20 | 页内动画提示默认不应画“先出现/关键结果”和数字徽章；这些标注常常新增遮挡且学习价值低。公式修复不能只处理被移动的 `graphicFrame`，有 fallback 预览的稳定公式也要在原框内重绘；目标框禁止 padding 外扩，否则会盖住旁边图片或文字。 |
| 2026-05-21 | 公式问题不能只看 bbox 不重叠；如果 fallback 图像本身被裁切、公式贴近文字、或多个公式在同一语义块内互相叠压，PDF 重绘仍会显得“挡住/乱码”。后续应把公式作为可读性对象处理：先校验公式预览完整性，再决定原位重绘、局部放大、移动同组对象或整组展开。|
| 2026-05-21 | 第 35 页这类底部拥挤区不能靠单对象硬挪；应允许公式/视觉对象按候选评分缩到 0.85/0.75/0.65 档，并对移动后的文本再做二次避让，避免“公式不挡了但说明文字互相压住”。|
| 2026-05-21 | 视觉输出不能用几何测试替代验收；只要截图中仍出现公式乱码、公式/文字互压、对象被推到标题区或页面整体更乱，就必须判定未修复。后端排版改动必须把渲染截图作为硬门禁，而不是只记录为“剩余风险”。|
| 2026-05-21 | 稳定公式不能无条件用 fallback 预览重绘；LibreOffice 原生渲染正常时重绘反而会造成拉伸和乱码。应先对最终 PDF 公式区域做渲染检测，只修检测失败的公式。|
| 2026-05-21 | 当复杂页对象级重排无法降低真实重叠时，应让质量门禁阻止落地；页底空间不足可采用整页轻微缩放，但渲染检测区域必须同步做同一缩放变换。|
| 2026-05-21 | 组合形状里的公式要区分“独立公式块”和“文字行占位公式”；带制表占位的行内公式不应被搬走，否则会把原本可读的题述行拆散。|
| 2026-05-21 | fallback 预览本身也可能是坏图；稳定公式只有在最终渲染失败且预览图通过可用性检查时才覆盖，不能用坏 fallback 去修正常原生渲染。|
| 2026-05-22 | 电路图、流程图这类由很多短线条/小图元拼成的语义图，不能按单个形状对象级重排；如果一次计划移动大量小图元且缺少长文本锚点，应判定为图元碎片化风险并保持原页关系。|
| 2026-05-22 | `a < r < b:`、`r > b` 这类短数学条件本质是单行条件文本，不是普通正文；LibreOffice 可能把它拆成多行，生成 guide deck 时应设置不换行并在安全空隙内扩宽文本框。|
| 2026-05-22 | AI 接入出现 `HTTP Error 401: Unauthorized` 时，根因通常是 API key、Base URL、Model 不属于同一服务商或 key 无效；后端不能把 urllib 原始异常裸抛给前端，应转成可操作提示，并把 `/v1` 根地址规范化为 `/v1/chat/completions`。|
| 2026-05-22 | AI 模型可能把要求的 JSON 对象返回成 JSON 数组；审计前必须先校验顶层类型，不能让 list/dict 进入 set 去重导致 `unhashable type: 'dict'`，prompt 也要明确“只返回一个 JSON 对象”。|
| 2026-05-22 | 给模型看的来源如果只有 `slide_text@p1#18` 这类短标，模型会照抄字符串，导致严格审计报 Invalid source ref；prompt 必须提供可复制的 JSON 来源，后端也应把短标规范化为 `{kind, slide, object_id}` 后再审计。|
| 2026-05-22 | AI 模型不一定严格按 schema 返回数组字段，可能把 `key_points`、`review_questions` 返回成字符串或对象；后端保存前要规范成数组，前端渲染也要用 `asList/asText` 防止 `items.forEach is not a function`。|
| 2026-05-22 | AI provider 读响应可能抛 `The read operation timed out`，这属于模型服务慢或网络不稳；后端应捕获 `TimeoutError/socket.timeout` 转成中文提示，并给生成类请求更长默认超时。|
