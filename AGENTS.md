# AGENTS.md

本仓库是“小鹏 AI 公开赛赛道 3”的 PPT 转 PDF 学习版项目。目标不是普通转换器，而是把 PPT 原生画面转换成更适合学习、复习、理解动画遮挡关系的 PDF。后续开发必须围绕这个目标，不要把路线带偏。

## 核心目标

1. `base.pdf` 保留原始转换结果，用于对照。
2. `guide.pdf` 是学习版成品，应在保留 PPT 基本页面和作者意图的基础上做局部微调、遮挡修复、动画展开和媒体关键帧替换。
3. 重排不是“抽出内容重新画一页”，而是“在原页面附近做可解释、可读、美观的局部修复”。
4. 比赛作品要像一个稳定产品：输出可看、逻辑可解释、样例可复现、失败原因可定位。

## 用户工作流偏好

1. 用简洁中文沟通，结论先行，不要写冗长背景。
2. 如果目标明确，直接执行，不要反复问指令。
3. 如果路线可能偏离最终愿景，必须主动纠正，并说明为什么。
4. 每次用户指出问题后，要更新 `lessons.md`。
5. 每次新任务开始前，先回顾 `lessons.md`。
6. 不能擅自开分支或新 worktree。
7. 不要做破坏性操作。涉及删除、重置、覆盖用户文件等高风险动作时先停下来确认。

## 必用流程

每次任务开始：

1. 读取适合当前任务的 skill，并在回复中列出本次会使用哪些 skill。
2. 读取 `lessons.md` 最近经验，避免重复踩坑。
3. 明确任务类型：
   - bug：先复现和定位根因，再写回归测试，再修。
   - 功能：先确认它是否符合项目愿景，再落地。
   - 大重构：先更新 `task_plan.md`、`progress.md`、`findings.md`。
   - 文档/UI 小改：可直接改，但仍要验证文件落地。

每次改动后：

1. 自己 Review 一轮：有没有路线偏移、硬编码样例、旧 job 验收、视觉负优化。
2. 跑针对性测试。
3. 影响后端转换时，必须重启 8765 服务，重新转换 `app/samples/test.pptx`。
4. 对影响视觉输出的改动，必须渲染最新 `guide.pdf` 截图检查。

## 路线防偏原则

1. 不要新增无实际价值的导读页来掩盖原页问题。
2. 不要把所有图片、公式、图形机械放到右侧或底部。
3. 没有遮挡关系、媒体关系、动画关系的视觉对象默认不动。
4. 有遮挡关系时，优先在原页面空白区局部修复；其次才考虑缩放、分拆、附近展开。
5. 图文关系优先于全局几何不重叠。公式、图、说明文字不能被拆散成无意义散点。
6. 几何不重叠不等于成功。还要看语义对应、阅读顺序、视觉美观、截图效果。
7. 所有“优化”都要能从评分、测试或截图中证明，不能只凭主观说法。

## 重排规则

1. 重排要保留 PPT 的基本页面感，不要重画成另一种课件。
2. 稳定对象不动。只移动遮挡对象、被遮挡对象、动画/媒体解释所需对象。
3. 文本框不要随意压缩高度，避免 PowerPoint/LibreOffice 产生文本外溢。
4. `graphicFrame`、OLE 公式默认保尺寸移动，不要轻易缩放。
5. 公式重排要注意 LibreOffice 可能使用 fallback 图，必要时同步 OOXML 坐标和 fallback 预览。
6. 标号不能盖住图、公式、文字；应放在对象外侧候选位。
7. 箭头要短、指向明确、留出安全距离，不要穿过正文或压住目标。

## GIF、视频和动态媒体规则

1. 不要假装 PDF 能保留 GIF/视频播放。学习版应转换为关键帧序列。
2. PPT 中的 GIF 可能只是普通图片关系，要从 `a:blip` 关系目标后缀识别。
3. 视频对象可能有封面图和 `a:videoFile`，解析时优先识别真实媒体关系。
4. 动态媒体的默认语义是“关键帧替换原媒体”，不是“保留封面再额外补图”。
5. 进入关键帧绘制时，必须先清完整 `source_rect`，再绘制评分选出的宫格或分离面板。
6. 小 GIF 可把第 1 帧放在原位，其余帧放附近空白区；大 GIF 可直接用宫格替代原媒体区。
7. 关键帧面板要避让标题、正文、图像，不得解决媒体可读性却破坏原页面信息。

## 评分布局规则

1. 布局不能靠单个固定位置。要生成候选位并打分。
2. 评分至少考虑：
   - 可读性：单帧尺寸、缩放比例。
   - 语义距离：离原对象越近越好。
   - 避让：标题、正文、已有图形、原媒体区。
   - 帧数：能表达过程，但不为帧数牺牲可读性。
   - 稳定性：不能让原本清晰的对象变差。
3. 测试应锁住评分结果的关键指标，例如 `mode`、行列数、cell 尺寸、是否清理原区域。
4. 如果截图表现差，优先改评分函数，不要写样例专用坐标。

## 测试与验收

常用命令：

```powershell
python -m unittest discover -s app\tests
python -m compileall app\backend
```

后端转换逻辑变更后必须执行：

```powershell
curl.exe -s -F "deck=@app/samples/test.pptx" http://127.0.0.1:8765/api/convert
```

验收规则：

1. 不能用旧 Web job、旧下载 PDF、旧截图判断新代码效果。
2. 后端代码变更后，先确认 8765 服务使用的是最新进程。
3. 真实转换 `app/samples/test.pptx` 后，打开最新 `outputs/<job_id>/guide.pdf`。
4. 视觉改动至少截图检查受影响页。
5. 第 1、2 页主要检查对象重排、公式、箭头、编号。
6. 第 3、4 页主要检查 GIF 关键帧是否替换原媒体、是否露出旧封面、是否压住标题或正文。
7. Web 报依赖错误时，先查服务进程实际 Python，再处理依赖。

## 样例与数据

1. `app/samples/test.pptx` 是核心回归样例，覆盖遮挡、公式、动画、GIF 等场景。
2. 不要捏造数据。生产代码不能用 Mock。
3. Mock 只能用于本地测试或明确的测试辅助，不得进入生产路径。
4. 生成物、缓存、临时输出不要纳入业务判断；以最新转换产物为准。

## 代码规范

1. 优先沿用现有模块和风格，不要随意引入新架构。
2. 新文件使用 UTF-8 无 BOM。
3. 手工改文件用 `apply_patch`。
4. 搜索优先用 `rg`；如果不可用或权限异常，再用 PowerShell `Select-String`。
5. 不要使用破坏性 git 命令。
6. 不要回滚用户已有改动；如果工作区很脏，只改任务相关文件。
7. 不要把 bug 用兜底逻辑藏起来。能暴露就暴露，能写测试就写测试。

## 输出习惯

1. 最终回复只写高信号内容：改了什么、验证了什么、输出在哪里、还有什么风险。
2. 多项内容用表格。
3. 不要复读已知背景。
4. 如果没跑测试或没验收截图，必须明说。
5. 结束时列出本次使用的 skill。
